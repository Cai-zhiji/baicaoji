# 百草计 — UX 流程访谈纪要 × Grill 决策

> 访谈日期：2026-07-10 | 版本：v2.0 | 状态：已实施

## 一、UX 流程访谈纪要（Grill Session）

以下记录了 2026-07-10 对百草计应用使用流程的系统性访谈。每个决策点包含问题、最终决策及理由。

### 1.1 核心流程决策

#### #1 搜索框与药方明细的滚动问题

**问题：** 开方页搜索框在上、药方明细在下，用户需要反复上下滚动来逐味添加药材，如何解决？

**决策：** 搜索框 sticky 固定在顶部，添加药材后自动滚动到药方明细底部（新加的那行）。

**理由：** 中医开方通常 10-15 味药，每次添加都需要在搜索和明细之间切换。sticky 搜索框 + 自动滚动消除了不必要的往返操作。

---

#### #2 病人是否必选

**问题：** 当前数据模型 patientId 为必填，但用户表示大多数是散客，不需要记录。

**决策：** 病人改为可选。数据库 patientId Int → Int?，API 不再要求 patientId，前端默认折叠病人选择区。

**理由：** 散客占就诊量 75%+。强制选病人增加了无意义的操作步骤。病人管理降级为「常客建档」功能。

---

#### #3 新建病人流程

**问题：** 当前使用 Modal Dialog，需要 3 步操作（点击按钮 → 填表单 → 点创建）。是否太繁琐？

**决策：** 改为一行内联创建：输入姓名 + 回车即可创建。同时处理 409 同名检测，自动选中已有病人。

**理由：** 开方时的思维不应被 Modal 打断。内联创建只需一个名字，性别默认「男」即可。

---

#### #4 药方模版的使用方向

**问题：** 是先选模版再调克数，还是先手动加药再补加载模版？

**决策：** 两个方向都有。因此模版区放在搜索框下方、药方明细上方，保证两个方向都顺手可达。

**理由：** 不同场景需要不同的操作顺序。模版加载时静默跳过已在药方中的药材，不覆盖已有克数。

---

#### #5 保存药方后的行为

**问题：** 清空药方、保留明细、还是跳转到详情页？

**决策：** 清空药方明细并重置病人选择（如果关联了病人）。

**理由：** 连续开方场景下，上一个药方的残留数据会干扰下一个病人。清空是最安全默认。模版功能已覆盖「复用」需求。

---

#### #6 导航架构

**问题：** 多少个底部标签？管理页面入口放在哪里？

**决策：** 3 个标签：开方、药方、库存。管理页面（药材/病人/模版/统计）入口放在药方和库存页面右上角。

**理由：** 管理操作低频（几天/几周一次），不应占主标签位。从相关页面进入管理更符合心智模型。

---

#### #7 库存不足处理

**问题：** 当前无提示，直接扣到负数。

**决策：** 保存前在前端标红提示库存不足的药材行（显示库存剩余量），但不阻断保存。

**理由：** 中医诊所的实际库存和系统库存常有偏差。提示即可，不应阻断操作。

---

### 1.2 信息架构决策

|------|----------|------|------|

### 2.2 数据模型变更

**Prescription 表的 patientId 字段从必填（Int）改为可选（Int?）：**

```
patientId Int?    // 之前: patientId Int
patient Patient?  // 之前: patient Patient
```

**影响：**

- API `POST /api/prescriptions` 不再要求 patientId
- API `GET /api/prescriptions` 返回的 patient 字段可能为 null
- 前端散客药方显示为「散客」，筛选时散客计入「全部」
- 需要运行 `npx prisma db push` 同步数据库

### 2.3 关键 UX 改进

- **病人可选**：散客无需建档，直接开方
- **内联创建病人**：输入姓名 + 回车，自动处理同名检测
- **搜索框 sticky**：添加药材时搜索栏始终可见
- **自动滚动**：添加药材后自动滚动到新行
- **库存警告**：保存前标红显示库存不足的药材
- **保存后清空**：连续开方场景下自动重置
- **管理入口内嵌**：药方页 → 模版/病人，库存页 → 药材/统计
- **液态玻璃层次**：轻（导航）/ 标准（面板）/ 重（模态框）三级
